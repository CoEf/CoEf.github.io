## 시작하며

"오브젝트 풀링은 Unity(C#)에서나 효과적이고, Godot은 내부에 캐싱 메커니즘이 있어서 풀링 효과가 떨어진다"는 이야기를 종종 듣는다. 결론부터 말하면 이 표현은 절반만 맞다. 정확히는 **Godot에는 Unity식 캐싱 메커니즘이 따로 있는 게 아니라, 애초에 C#/Unity가 쓰는 추적형 가비지 컬렉터(GC) 자체가 없다.** 이 차이 때문에 풀링이 해결하는 문제의 성격과 체감 효과의 크기가 달라지는 것이다.

세 가지 질문으로 나눠서 정리해본다.

1. C# 가비지 컬렉터는 왜 할당/해제를 반복할 때 프레임 드랍을 일으키는가
2. Godot은 어떤 메커니즘으로 효율적인 할당/해제를 하는가
3. Godot에서 가비지 컬렉터를 대체하는 방식은 무엇인가

## 1. C# 가비지 컬렉터가 프레임 드랍을 일으키는 이유

C#(.NET/Mono)의 GC는 **추적형(tracing) 가비지 컬렉터**이며, 세대별(generational) 마크 앤 스윕(mark-and-sweep) 방식으로 동작한다.

- 새로 할당된 객체는 가장 작은 세대인 Gen0 영역에 들어간다.
- Gen0 영역이 꽉 차면 GC가 "지금 살아있는 객체가 무엇인지" 참조 그래프를 따라가며 찾아내고(mark), 더 이상 참조되지 않는 객체가 차지한 공간을 회수한다(sweep).
- 이 마크 단계에서는 도달 가능한 객체를 전부 방문해야 하므로, 이 작업이 진행되는 동안 스레드를 멈춰야 하는 구간(stop-the-world)이 발생한다.
- `Update()`나 매 프레임 실행되는 로직 안에서 `new`로 임시 객체(총알, 이펙트, 임시 리스트 등)를 계속 생성하면 Gen0가 빠르게 차오르고, GC가 더 자주 트리거된다. 트리거될 때마다 stop-the-world 구간이 생기고, 이게 곧 프레임 스파이크로 체감된다.

여기에 힙 단편화(fragmentation) 문제도 더해진다. 크기가 제각각인 객체를 계속 할당·해제하면 빈 공간이 조각조각 나뉘고, 이후엔 압축(compaction) 비용까지 추가로 발생한다.

즉 Unity에서 오브젝트 풀링을 쓰는 진짜 목적은 "할당 속도 자체를 높이기 위해서"가 아니라, **GC가 추적해야 할 객체 수와 할당 빈도를 최소화해서 GC 사이클이 애초에 덜 발생하도록 만드는 것**이다. 풀링의 핵심 효과는 할당 속도 개선이 아니라 **GC 트리거 억제**에 있다.

## 2. Godot은 어떤 방식으로 할당/해제를 처리하는가

여기서 흔한 오해를 정정할 필요가 있다. Godot 코어에는 Unity의 GC를 대체할 만한 범용 오브젝트 캐시나 슬랩(slab) 할당자가 별도로 존재하는 게 아니다. Godot의 `memnew()` / `memdelete()` 매크로는 결국 표준 `malloc`/`free`를 감싸는 래퍼이고, Godot 엔진 프로포절 트래커에는 이걸 mimalloc 같은 고성능 할당자로 교체하자는 논의가 여전히 진행 중이다. 다시 말해 현재 기본값은 특별한 캐싱 계층이 아니라 시스템 기본 `malloc`이다.

Godot이 실제로 갖고 있는 최적화는 다음 두 가지에 가깝다.

- **COW(Copy-on-Write) 기반 컨테이너** — `Array`, `String`, `PackedArray` 계열은 복사 시 실제 데이터를 복사하지 않고 참조 카운트만 증가시켰다가, 실제 수정이 일어날 때만 데이터를 복사한다. 재할당 빈도를 줄여주는 최적화이지, 오브젝트 풀링을 대체하는 개념은 아니다.
- **큰 데이터 전용 동적 메모리 풀(3.x 시절의 PoolVector)** — 이미지, 메시 지오메트리처럼 큰 데이터를 다룰 때만 쓰던 별도 풀 구조였는데, Godot 4에서는 COW 기반 `Vector`로 대체되면서 이 개념은 대부분 사라졌다. 총알이나 이펙트 인스턴스 같은 일반 스크립트 레벨 오브젝트에는 해당되지 않는다.

즉 "Godot이 내부적으로 오브젝트를 캐싱해줘서 풀링이 덜 중요하다"는 표현은 정확하지 않다. 실제로는 **애초에 GC 트리거라는 개념 자체가 없기 때문에, 풀링이 해결하려는 문제의 성격이 다르다**고 보는 게 맞다.

## 3. Godot에서 가비지 컬렉터를 대체하는 방식

정답은 **참조 카운팅(Reference Counting)과 수동 메모리 관리의 혼합**이다.

- `RefCounted`를 상속하는 클래스(예: `Resource`, 그리고 상속을 명시하지 않은 기본 GDScript 클래스)는 내부 참조 카운터가 0이 되는 즉시 메모리에서 해제된다. 지연도 없고 별도의 스캔 사이클도 없다. 멀티스레드 환경에서도 안전하도록 원자적(atomic) 연산으로 카운터를 관리한다.
- `Node`나 `Object`처럼 `RefCounted`를 상속하지 않는 클래스는 자동으로 해제되지 않는다. `free()` 또는 (Node의 경우) `queue_free()`를 명시적으로 호출해야 메모리에서 사라진다.

이 방식의 핵심 장점은 **결정론적(deterministic) 해제**다. 참조가 끊기는 순간 즉시 해제되므로 "GC가 언제 돌지 모른다"는 불확실성 자체가 없고, 그래서 Unity식 stop-the-world 스파이크가 애초에 발생하지 않는다.

## 그래서 Godot에서 오브젝트 풀링은 의미가 없는가

아니다. 여전히 의미가 있다. 다만 **풀링으로 얻는 이득의 종류가 다르다.**

| 구분 | C# / Unity | GDScript / Godot |
|---|---|---|
| 풀링 없이 반복 할당/해제 시 문제 | GC 트리거 빈도 증가 → stop-the-world 스파이크 | GC 스파이크는 없음. 대신 `malloc`/`free` 호출 비용 + 생성자/소멸자 비용 + (Node라면) 씬 트리 진입/이탈, 시그널 연결/해제 비용이 매 프레임 누적 |
| 풀링으로 얻는 효과 | 프레임 스파이크 억제. 효과가 크고 체감이 확실함 | 위 비용들의 누적을 줄임. 스파이크성보다는 평균 프레임 타임 개선에 가까움 |

`Node` 기반 인스턴스(총알, 이펙트 등)를 재사용하는 오브젝트 풀에서 얻는 이득은 GC 회피가 아니라 **`instantiate()` 비용, 씬 트리 진입(`add_child`)/이탈 비용, 초기화 로직 재실행 비용을 건너뛰는 것**에서 나온다. Unity만큼 "안 하면 프레임이 뚝뚝 끊긴다" 수준의 극적인 체감 효과는 아니지만, 발사 빈도가 높은 총알·파티클 시스템에서는 여전히 유의미한 최적화다.

## 정리

- C# GC의 프레임 드랍은 추적형 마크 앤 스윕 방식이 만드는 stop-the-world 구간 때문이며, 풀링은 이 GC 트리거 자체를 억제하는 역할을 한다.
- Godot 코어에는 오브젝트 전용 캐싱 할당자가 따로 있는 게 아니라, 표준 `malloc`/`free`를 매크로로 감싼 구조에 COW 컨테이너 최적화가 더해진 정도다.
- Godot이 GC를 안 쓰는 이유는 캐싱이 뛰어나서가 아니라, **참조 카운팅(RefCounted) + 수동 관리(Node/Object)**로 결정론적 메모리 해제를 하기 때문이다.
- 따라서 Godot의 오브젝트 풀링은 GC 스파이크 방지가 아니라 인스턴스 생성/초기화/씬 트리 진입 비용 절감이 목적이며, 여전히 유효한 최적화 기법이다.
